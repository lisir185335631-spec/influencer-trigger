# Alembic Migrations

Schema changes are managed by Alembic. Do **not** add ALTER TABLE statements to
`main.py` lifespan.

## Common commands

```bash
cd server
alembic current                              # current revision
alembic history                              # full migration history
alembic revision --autogenerate -m "desc"   # detect model changes, generate migration
alembic upgrade head                         # apply all pending migrations
alembic downgrade -1                         # roll back one step
alembic stamp head                           # mark existing DB as head (no-op migrations)
```

## Adding a new field — standard workflow

1. Edit `server/app/models/<name>.py` and add the column.
2. `cd server && alembic revision --autogenerate -m "add <field> to <table>"`
3. Review the generated file in `alembic/versions/` — confirm `op.add_column` looks correct.
4. `alembic upgrade head`
5. Commit the migration file.

## Onboarding an existing database (no alembic_version table)

```bash
cd server
alembic stamp head
```

This marks the DB as already at the latest revision without running any DDL.

## SQLite limitations

SQLite does not support `DROP COLUMN`, constraint changes, etc. Use batch mode in
migration files for these operations:

```python
with op.batch_alter_table("table_name") as batch_op:
    batch_op.drop_column("col")
```

The `env.py` already sets `render_as_batch=True` globally, so autogenerated
migrations will use batch mode automatically.

## Do NOT

- Add `ALTER TABLE` to `main.py` lifespan.
- Manually edit the `alembic_version` table.
- Delete migration files that have already been applied to production.
